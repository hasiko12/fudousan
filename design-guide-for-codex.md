# UI/UX デザイン指示書 — 反響統合・マッチング・内見調整コパイロット

対象: Codex（実装担当）
前提仕様: `mvp-a-real-estate-copilot-spec.md`
スタック: Next.js App Router + TypeScript + Tailwind CSS + shadcn/ui + Supabase
参考実装: `reference-ui-mockup.html`（ダッシュボード／リード詳細の見た目とインタラクションの“答え”。色・余白・状態表現はこのHTMLを正とする）

この文書は「何を作るか（仕様書）」ではなく「どう見せ、どう触らせるか」を定義する。仕様書とこの指示書が矛盾した場合、業務要件は仕様書、見た目と操作感はこの指示書を優先する。

---

## 0. デザインの芯（これだけは外さない）

このプロダクトは消費者向けアプリではなく、**1人〜数人の不動産担当者が反響を取りこぼさず素早くさばくための業務デスク**である。だから:

1. **派手さより速さと密度。** ヒーローや装飾は不要。1画面に「いま何をすべきか」と「次の1クリック」を出す。
2. **AIと人間の境界を“見える形”にする。** これがこのアプリの最重要かつ唯一の主役的デザイン要素（=シグネチャ）。AIが書いたものは必ず紫みのパネル＋「送信前に確認」表示＋承認ゲートを通す。人間が承認していないものが顧客に出ていく見た目を**絶対に作らない**。
3. **断定しない言葉づかいを UI が支える。** 空室・金額・契約可否は「確認のうえご案内」というトーン。確信度が低い項目は黄色で「要確認」と出す（推測を断定に見せない）。
4. **取りこぼさない安心感。** ステータスと期限が一覧で“ひと目”で分かる。期限超過は赤で必ず目立たせる。

> シグネチャ要素 = **AIドラフトの質感**。紫(`--ai`)のティント面＋スパークアイコン＋確信度チップ＋承認ボタン。ここにだけ色とコントラストを使い、他は静かに保つ。

## 1. デザイントークン

CSS 変数 or `tailwind.config.ts` の `theme.extend.colors` にそのまま入れる。HTMLモックアップと同一値。

### 1.1 色

```css
:root{
  /* 面・文字 */
  --paper:#F4F6F9;        /* アプリ背景（純白にしない） */
  --surface:#FFFFFF;      /* カード */
  --surface-2:#FBFCFD;    /* テーブル偶数行・hover */
  --ink:#17212E;          /* 本文 */
  --ink-2:#46505F;        /* 副次 */
  --ink-3:#7A8595;        /* 補助・ラベル */
  --line:#E5E9EF;         /* 罫線 */
  --line-strong:#D4DAE3;  /* 入力枠 */

  /* ブランド／一次アクション（信頼の青） */
  --primary:#2557C7; --primary-700:#1C449E; --primary-tint:#EAF0FC;

  /* AIアクセント（=シグネチャ。AI生成物にだけ使う紫） */
  --ai:#6B4FD8; --ai-700:#5238B8; --ai-tint:#F2ECFE; --ai-line:#DCD2F7;

  /* セマンティック */
  --ok:#1C875A;     --ok-tint:#E6F4EC;     /* 成功・確定 */
  --warn:#A9701A;   --warn-tint:#FBF1DD;   /* 確信度低・注意 */
  --danger:#C24034; --danger-tint:#FBEAE7; /* 期限超過・失注・権限エラー */
  --teal:#0E6E76;   --teal-tint:#E2F2F3;   /* 接触済ステータス */
}
```

ルール:
- **紫(`--ai`)はAI生成物以外に使わない。** ボタンや装飾で安易に使うとシグネチャが死ぬ。
- 背景は `--paper`。カードは `--surface`。純白の上に純白を重ねない。
- 赤(`--danger`)は「期限超過 / 失注 / 権限エラー」だけ。多用しない。

### 1.2 ステータスのカラーマップ（リードの8状態）

バッジは「左に色ドット＋ラベル」で統一（`reference-ui-mockup.html` の `.badge-status` 参照）。

| status | 日本語 | bg | text / dot |
| --- | --- | --- | --- |
| `new` | 未対応 | `--primary-tint` | `--primary` |
| `contacted` | 接触済 | `--teal-tint` | `--teal` |
| `matching` | 物件提案中 | `--ai-tint` | `--ai` |
| `viewing_scheduling` | 内見調整中 | `--warn-tint` | `--warn` |
| `viewing_confirmed` | 内見確定 | `--ok-tint` | `--ok` |
| `applied` | 申込/買付 | `#DFF0E6` | `#14663F`（太字） |
| `lost` | 失注/終了 | `#F0EDED` | `#8A6F6B` |
| `archived` | 保留 | `#EEF0F3` | `#6B7585` |

`urgency=high` は別途「急ぎ」ピル（`--danger-tint` / `--danger`）。期限超過は赤太字で残り時間/超過を出す。

### 1.3 タイポグラフィ

- フォント: `"Inter","Noto Sans JP",-apple-system,"Hiragino Kaku Gothic ProN","Yu Gothic",sans-serif`
- **数値（金額・時刻・スコア・KPI）は必ず `font-variant-numeric: tabular-nums`**（桁が揃って業務ツールらしくなる）。`.num` クラスで適用。
- 装飾的なセリフ体・極端なディスプレイ体は**使わない**（可読性と業務トーンを優先する意図的な抑制）。
- スケール:

| 用途 | size / weight |
| --- | --- |
| KPI数値 | 28 / 700 |
| 顧客名（詳細ヘッダ） | 18 / 700 |
| ページ/カード見出し | 13 / 600 |
| 本文 | 14 / 400 |
| 補助・メタ | 12 / 400, `--ink-3` |
| エディタ本文（返信文） | 13 / line-height 1.85 |
| セクション・アイブロウ | 11 / 600, tracking .04em, 左に3pxの縦バー（青、AI文脈は紫） |

### 1.4 形・影・余白

- 角丸: カード `12px` / 入力・ボタン `8px` / バッジ `6–7px` / ピル `999px`
- 影: カードは極薄 `0 1px 2px rgba(20,30,45,.04), 0 1px 3px rgba(20,30,45,.07)`。ポップオーバー/モーダルのみ濃い影。
- 余白: 4の倍数。コンテンツpadding `22px`、カード内 `16px`、要素間 `18px`。
- 罫線で区切る。影でフロートさせすぎない（フラットで密に）。

## 2. アプリシェル

```
┌──────────┬─────────────────────────────────────────┐
│          │  topbar: パンくず | [反響を貼り付け]      │
│ sidebar  ├─────────────────────────────────────────┤
│ (固定)   │                                          │
│          │  content (max-width なし・流動)           │
└──────────┴─────────────────────────────────────────┘
```

- **左サイドバー（固定 232px, 濃色 `#10192A`）**: ダッシュボード / リード / 物件 / 受信トレイ / 設定。各項目に件数バッジ。現在地は `--primary` でハイライト。最下部に担当者名＋ロール。
- **トップバー（高さ56px, 白）**: 左にパンくず、右に常設の主要アクション **「反響を貼り付け」**（`--primary`）。反響登録はどの画面からも1クリックで開ける（速さが命）。
- レスポンシブ: 1000px以下でサイドバーを上部のドロワー/ハンバーガーに。詳細画面の2カラムは1カラムに落とす。

## 3. 画面別レイアウト

### 3.1 `/dashboard` — 「今日やること」を最初に出す

上から: **KPIカード4枚 → 未対応リード表 ＋ 今日のタスク（2カラム）**。

- KPIは「未対応 / 内見調整中 / 返信文未承認 / 期限超過」。各カードは左端3pxの色レール＋大きな数値＋サブ行。**期限超過カードだけ数値を赤**にして視線を集める。
- 未対応リード表: 顧客（名前＋物件/条件メタ）/ 反響元 / ステータスバッジ / 経過時間。**経過が長い＝赤**。行クリックで `/leads/[id]` へ。
- 今日のタスク: チェックボックス＋タイトル＋（期限・関連リード）。期限超過は赤。完了は取り消し線＋緑チェック。
- KPIは「数字を見せる」のが目的ではなく「タップして対象一覧に飛ぶ」導線。クリックで該当フィルタ済みの `/leads` に遷移させる。

### 3.2 `/leads` — 一覧と仕分け

- 上部: 検索 + ステータスフィルタ（タブ or セグメント）+ 反響元フィルタ + 担当者フィルタ。
- 表: 顧客 / 反響元 / ステータス / 担当 / 次アクション期限 / 最終更新。デフォルトは**最終更新の新しい順**だが、未対応かつ期限が近いものを上に寄せる重み付けを推奨。
- 一括ステータス変更（チェックボックス選択 → 上部バー）。CSVエクスポートは **ownerのみボタンを表示**（staffには出さない／押すと権限エラー文言）。

### 3.3 `/leads/[id]` — このアプリの心臓部

仕様7ステップ（要約→候補物件→内見候補→返信文→タスク）を**上から下への作業フロー**として配置する。`reference-ui-mockup.html` のレイアウトを正とする。

```
[ ヘッダ: 顧客名 + ステータス | 連絡先 | 急ぎ |  条件再抽出 / マッチング再実行 / 返信文を生成 ]
┌────────────────┬──────────────────────────────────────────┐
│ 左レール(300px) │ 作業フロー(流動)                            │
│ ・希望条件      │ ① AI要約パネル（紫・要確認）                │
│  （確信度チップ）│ ② AI物件マッチング（候補カード×3, スコア+理由）│
│ ・ステータス/   │ ③ 内見候補（候補スロット×3）                │
│   次アクション   │ ④ AI返信文ドラフト（紫・承認ゲート）         │
│  (sticky)       │ ⑤ 確認タスク                               │
│                 │ ⑥ 活動履歴                                 │
└────────────────┴──────────────────────────────────────────┘
```

- **左レール = 動かない事実**（顧客の希望条件・ステータス）。スクロールしても sticky で残す。
- **右 = 操作する作業**。上から順にこなせば反響対応が完結する設計。
- ヘッダの3アクション（再抽出/再マッチ/返信生成）は常に手の届く位置に。

### 3.4 `/properties` — 15件の物件管理

- カードグリッド or 密なテーブル（15件規模なのでテーブルで十分）。公開/非公開トグル、条件タグ（ペット可・駐車場・駅近など）をチップで表示。
- 編集はモーダル。CSVインポートボタン。

### 3.5 `/inbox` — 反響の取り込み

- 左: 反響本文の貼り付けフォーム＋反響元セレクト＋受信日時。右: **AI解析プレビュー**（抽出された顧客情報・条件をリアルタイム表示。確信度の低い項目は黄色）。
- 「この内容で登録」ボタン。取込失敗は別タブ/下部に失敗キューとして残し、再試行できる。

### 3.6 `/settings`

- 担当者 / 反響元 / 返信テンプレート / 営業時間 / 内見可能時間テンプレート / AI設定。素直なフォーム群でよい。

## 4. 主要コンポーネント仕様

### 4.1 AIドラフトパネル（シグネチャ）★最重要

AI要約・返信文ドラフトはこの見た目で統一する（`.ai-panel` / `.draft-box`）:

- 枠 `--ai-line`、背景は `--ai-tint` の薄いグラデ。
- 上部に**リボン**: スパークアイコン＋「AI◯◯ ・ 送信前に確認してください」（`--ai-700`）。
- 確信度が低い/注意点があれば、パネル内に黄色の `ai-disclaimer`（「エリアの確信度が低めです」等）。
- **承認ゲート（返信文のみ）**: フッタに鍵アイコン＋「承認するまで顧客には送信されません」。右に [破棄] [編集] [承認する]。承認後は「✓ 承認済み・手動送信できます」（緑）に置き換え、自動送信ボタンは存在させない。
- チャネル切替（メール/LINE/SMS）はピル型タブ。LINE版は短文、メール版は丁寧、という長さの差が見た目でも分かるように。

### 4.2 確信度チップ（confidence）

- `high`: チップを出さない（ノイズになる）。
- `medium`: グレーの「普通」チップ（控えめ）。
- `low`: **黄色「要確認」**（`.conf-low`、頭に「!」の丸）。AIが推測した・本文に無かった項目に必ず付ける。値が無いものは `--ink-3` で「未確認」。

### 4.3 物件候補カード

- 左に物件名＋メタ（住所・駅徒歩・間取り・賃料＋管理費）、右に**大きなスコア数値**（`--primary`）。
- 下段に3種の理由行をアイコンで色分け:
  - 緑チェック = 推薦理由（予算内・駅近 等）
  - 黄三角 = 懸念点（ペット未確認・入居日が遅い 等）
  - 紫はてな = 顧客に確認すべき質問
- 候補0件のときは空状態（後述）で「確認すべき質問」だけを出す。

### 4.4 内見候補スロット

- 3枚のスロット（曜日・時刻・集合場所）。クリックで選択トグル（`.pick`、青）。「候補であって確定ではない」ことをカード見出しに明記（`候補のみ・確定前にチェック必須`）。確定操作は別途チェックリストを通す。

### 4.5 タスク行

- チェックボックス＋タイトル＋メタ（自動生成 / 対象物件 / 期限）。完了は緑チェック＋取り消し線。期限超過は赤メタ。

### 4.6 活動履歴

- 縦タイムライン。AIアクションはドットを紫、人間の操作はグレー。時刻表示。`detail` jsonbの中身は人間が読める日本語に整形して出す。

### 4.7 ボタン階層

- `primary`（青塗り）= その画面の主操作（反響を貼り付け / 登録 / ステータス更新）。1画面に1つを基本に。
- `ai`（紫塗り）= AI生成トリガ（返信文を生成 等）だけ。
- `default`（白＋枠）= 副次。`ghost`（枠なし）= 三次（すべて見る 等）。

## 5. 状態（空・読み込み・エラー）— 手を抜かない

仕様19.2のテスト観点に直結。空/失敗で画面を壊さない。

- **物件0件**: `/properties` と候補欄に空状態カード。「物件を登録すると候補を提案できます」＋[物件を追加]。
- **マッチ候補0件**: 物件カード欄に「条件に合う物件がありませんでした」＋AIが出した**確認質問だけ**を表示（仕様UC-03）。
- **AI抽出/生成中**: スケルトン（パネル形のグレー脈動）。完了まで承認ボタンは disabled。
- **AI APIエラー**: パネルを赤枠にし「生成に失敗しました。もう一度お試しください」＋[再実行]ボタン（仕様19.2）。抽出失敗でも**リード自体は登録済みのまま**にし、リード詳細から再抽出できる導線を残す。
- **電話/メールが null**: フィールドは「未登録」とグレーで表示し、欠落で行が崩れないようにする。
- **権限エラー（staffがCSVエクスポート等）**: 赤系トースト「この操作には owner 権限が必要です」。エラーは謝らない・曖昧にしない・次の手を示す。

文言は仕様書のトーンに合わせ、システム用語を出さない（「webhook」ではなく「メール転送」など、利用者の言葉で書く）。

## 6. 個人情報・安全の UI 反映

- 画面上のエラーメッセージ・トーストに**電話番号/メール/問い合わせ本文を出さない**（仕様12.2）。
- CSVエクスポート・完全削除の導線は owner のみ表示。削除は基本「アーカイブ」、完全削除は確認ダイアログ（取り消せない旨を明記）。
- リード削除は即時実行しない。アーカイブを既定の動作にする。

## 7. アクセシビリティ／品質フロア（暗黙の最低ライン）

- すべての操作要素にキーボードフォーカスリング（可視）。モーダルはフォーカストラップ＋Escで閉じる。
- 色だけに意味を持たせない（ステータスは色＋ラベル＋ドット、確信度は色＋テキスト＋アイコン）。
- コントラスト比 4.5:1 以上（本文）。`--ink-3` は補助テキスト専用で本文に使わない。
- `prefers-reduced-motion` を尊重。アニメは「承認時の置き換え」「行hover」程度に抑える。過剰な動きは入れない。
- 数値はタブ揃え、日時は `YYYY/MM/DD HH:mm` 統一。

---

## 8. 文言（マイクロコピー）の方針

- ボタンは「起きることそのもの」を書く: 「送信」ではなく文脈に応じて「承認する」「ステータスを更新」「物件を追加」。
- AI由来の表現は断定を避ける: 「空室です」ではなく「空室状況を確認のうえご案内」。返信文ドラフト内も同様（仕様7.7の禁止事項に従う）。
- 空画面は誘い文句にする:「物件を登録すると、反響に合う候補を自動で提案できます。」

## 9. 実装優先（見た目の観点）

仕様14のPhase 0に合わせ、見た目もこの順で固める:

1. **トークン＋アプリシェル**（sidebar/topbar、色・フォント・バッジを共通化）。
2. **ステータスバッジ・確信度チップ・KPIカード・テーブル**の共通コンポーネント化（最初に作ると後が速い）。
3. **`/leads/[id]` の作業フロー**（特に AIドラフトパネルと承認ゲート＝シグネチャ）。
4. ダッシュボード → 一覧 → 物件 → 受信トレイ → 設定。

shadcn/ui を使う場合: Badge / Card / Button / Table / Tabs / Dialog / Skeleton / Toast を上記トークンで上書きし、AIパネルだけは専用コンポーネント（`<AiDraftPanel>` `<ConfidenceChip>` `<StatusBadge>`）を新規作成する。

## 10. やってはいけないこと（チェックリスト）

- [ ] AIが書いた文章を、承認前なのに「送信済み」風に見せる。
- [ ] 自動送信ボタンを置く（このMVPに存在してはいけない）。
- [ ] 紫(`--ai`)を装飾やAI以外のボタンに流用する。
- [ ] 空室・金額・契約可否を断定する文言を画面/ドラフトに出す。
- [ ] 確信度の低い抽出値を、確定値と同じ見た目で出す。
- [ ] エラー/トーストに個人情報を載せる。
- [ ] 純白背景＋セリフ体＋テラコッタ等の「いかにもテンプレ」な見た目にする。装飾より密度と速度。
- [ ] 物件0件・候補0件で画面が崩れる/真っ白になる。

### 付録: 参考実装の使い方

`reference-ui-mockup.html` をブラウザで開くと、ダッシュボードとリード詳細を切り替えられる（右上トグル）。承認ボタン・チェックボックス・スロット選択は動作デモ。**色・余白・状態表現・AIパネルの質感はこのHTMLをピクセルの正**として実装してよい。コンポーネント分割と本番データ接続だけ Codex 側で行う。
